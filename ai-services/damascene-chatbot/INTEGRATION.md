# Damascene Chatbot — Backend Integration Guide

This guide covers everything the Laravel backend needs to wire up the Damascene Art chatbot RAG service. Hand it to whoever (human or AI) is doing the integration.

---

## 1. What this service does

A FastAPI service that takes a user message + recent conversation, retrieves relevant Q&A from a 2,035-pair knowledge base via ChromaDB vector search, and returns a bilingual (Arabic / English) reply generated by an LLM. The Laravel backend acts as a proxy: frontend → Laravel → chatbot service → Laravel → frontend.

The service is **stateless**. It does **not** store conversations. The backend owns chat history.

---

## 2. Service location

- Local dev: `http://localhost:8000`
- Production: whatever host you deploy to. Set it in Laravel as `CHATBOT_SERVICE_URL`.
- Transport: HTTP/JSON. Responses are buffered server-side and returned as a single JSON document — there is no SSE/streaming exposed to the backend.

---

## 3. Authentication

The chatbot service supports an **optional shared-secret API key** via the `X-API-Key` header.

| State | Behavior |
|-------|----------|
| Service env has `API_KEY=<value>` | Auth is **enforced** on `/chat`, `/ingest`, `/search`. Missing/wrong header → `401`. |
| Service env has `API_KEY=` (empty) | Auth is **disabled**. Endpoints accept any caller. |
| `/health` | **Always open**, no auth. Use for liveness/readiness probes. |

**Operational rule:** turn auth on in any environment reachable from outside the local machine. Generate a long random value (e.g. `openssl rand -hex 32`) and put the same value on both sides:

- Chatbot service `.env`: `API_KEY=<value>`
- Laravel `.env`: `CHATBOT_API_KEY=<same value>`

## 4. Endpoints

### `POST /chat` — main endpoint

**Request:**
```http
POST /chat HTTP/1.1
Content-Type: application/json
X-API-Key: <required when auth is on>

{
  "message": "هل تشحنون للخارج؟",
  "session_id": "user-42-conv-7",
  "conversation_history": [
    {"role": "user", "content": "مرحبا"},
    {"role": "assistant", "content": "أهلاً بكم في Damascene Art..."}
  ]
}
```

**Field contract:**
| Field | Type | Required | Rules |
|-------|------|----------|-------|
| `message` | string | yes | 1–500 chars. Arabic, English, or mixed. |
| `session_id` | string | yes | Any non-empty string. Used only for correlating logs; the service does not persist it. |
| `conversation_history` | array | no | Max 20 items. Each item: `{"role": "user"\|"assistant", "content": string}`. Items missing either field are silently dropped. |

**Response (200):**
```json
{
  "response": "نعم، نوفّر خدمة الشحن إلى جميع أنحاء العالم...",
  "session_id": "user-42-conv-7"
}
```

**Error responses:**
| Status | Body | Meaning |
|--------|------|---------|
| 401 | `{"detail": "Invalid or missing API key"}` | Auth is on; header is wrong/missing. |
| 422 | `{"detail": [...]}` (Pydantic shape) | Body schema invalid (message empty/too long, missing session_id, etc.). |
| 500 | `{"detail": "Internal server error"}` | Unexpected exception. Check chatbot service logs for the traceback. |

**Latency:** typically 3–10 s. A long, multi-paragraph Arabic answer can reach 20–30 s. Set your HTTP client timeout to **≥ 60 s**.

---

### `GET /health` — liveness check

No auth, no body.

**Response (200):**
```json
{
  "status": "ok",
  "kb_size": 2035,
  "model": "gemini-3.1-pro"
}
```

If `kb_size` is `0`, the knowledge base hasn't been ingested yet — the bot will give vague answers until ingestion runs.

---

### `POST /ingest` — admin only

Triggers full re-ingestion of the KB into ChromaDB. **Synchronous, ~15 minutes.** The backend should never call this on a user request — it's an operational endpoint for ops/maintenance scripts. Requires `X-API-Key` when auth is on.

### `POST /search?query=...&top_k=5` — debug only

Returns the top-K KB hits for a query without calling the LLM. Useful when a particular customer question is getting irrelevant answers and you want to know what the retriever pulled. Requires `X-API-Key` when auth is on. Not part of the user-facing flow.

---

## 5. Conversation history — important contract

- The service is **stateless**. The backend is the source of truth for chat history.
- Persist each turn (user + assistant) keyed by your own `session_id` (e.g. a `chat_messages` table). On each new user message:
  1. Load the last ~10 turns for that session from your DB.
  2. Send them as `conversation_history`.
  3. After getting the response, append both the user message and the response to your DB.
- If you omit `conversation_history`, the bot has no memory — it answers each message in isolation.
- The service trims internally to the last 30 messages (~15 turns), so sending more than that wastes bandwidth.

---

## 6. CORS

This service is intended for **server-to-server** calls (Laravel → chatbot). Browsers should never call it directly in production.

If a browser ever does need to call it (e.g. an internal admin UI), the operator must add that origin to the chatbot's `CORS_ORIGINS` env var (comma-separated). The default is `*` for development convenience and must be tightened in production.

## 7. Laravel-side wiring

### `.env`
```
CHATBOT_SERVICE_URL=http://chatbot.internal:8000
CHATBOT_API_KEY=<same value as the chatbot service's API_KEY>
```

### `config/services.php`
```php
'chatbot' => [
    'url'     => env('CHATBOT_SERVICE_URL', 'http://localhost:8000'),
    'api_key' => env('CHATBOT_API_KEY', ''),
],
```

### `app/Services/ChatbotService.php`
```php
namespace App\Services;

use Illuminate\Support\Facades\Http;

class ChatbotService
{
    public function chat(string $message, string $sessionId, array $history = []): array
    {
        $url    = config('services.chatbot.url');
        $apiKey = config('services.chatbot.api_key');

        // 60s because the RAG service streams the LLM internally and long
        // answers can take 20–30s to fully buffer before returning.
        $request = Http::timeout(60)->acceptJson()->asJson();

        if (! empty($apiKey)) {
            $request = $request->withHeaders(['X-API-Key' => $apiKey]);
        }

        $response = $request->post("{$url}/chat", [
            'message'              => $message,
            'session_id'           => $sessionId,
            'conversation_history' => $history,
        ]);

        if ($response->failed()) {
            logger()->error('Chatbot call failed', [
                'status' => $response->status(),
                'body'   => $response->body(),
            ]);
            return [
                'response'   => 'نعتذر، حدث خطأ تقني. يرجى المحاولة مرة أخرى.',
                'session_id' => $sessionId,
            ];
        }

        return $response->json();
    }
}
```

### `routes/api.php`
```php
Route::post('/chat', [ChatController::class, 'send'])
    ->middleware(['throttle:10,1']);
```

The chatbot service has **no built-in rate limit**. Always rate-limit on the Laravel side.

## 9. Troubleshooting

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| `401 Invalid or missing API key` | Header missing or values diverged | Verify `CHATBOT_API_KEY` (Laravel) == `API_KEY` (chatbot service) |
| `422 Unprocessable Entity` | Bad request body | Check message ≤ 500 chars, `session_id` present, history items have `role` + `content` |
| `500 Internal server error` | Upstream LLM proxy failure or DB issue | Check chatbot service logs (now structured: `timestamp [LEVEL] module: message` + tracebacks for exceptions) |
| `Connection refused` | Chatbot service not running | On the chatbot host: `python scripts/run.py` |
| Timeout from Laravel | Reply was longer than client timeout | Set `Http::timeout(60)` or higher |
| Bot replies generically / off-topic | KB not ingested | Hit `/health` — if `kb_size` is 0, run `python scripts/ingest.py` on the chatbot host |
| Bot has no memory between turns | `conversation_history` not being sent | Confirm Laravel is loading and forwarding prior turns |

---

## 10. Things to **not** do

- Do **not** call `/ingest` from a user-facing request path — it's a 15-minute admin operation.
- Do **not** put the LLM provider key (`LLM_API_KEY`) in the Laravel `.env`. Only the chatbot service needs it.
- Do **not** use user-supplied content directly as `session_id`. Generate it server-side (UUID or signed combination of user_id + conversation_id).
- Do **not** log full message bodies or model responses to shared/customer-visible logs in production (PII risk).
- Do **not** disable Laravel-side rate limiting. The chatbot service has none.
